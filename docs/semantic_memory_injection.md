# Semantic Memory Injection
## Ambient Context Architecture for Large Language Models

*Concept Document v1.0 — February 2026*

---

**Origin:** Derived from first-principles reasoning about the nature of human cognition vs. transformer-based language model inference. Building on earlier work in the Cognitive Team Architecture and Semantic Token Passthrough concepts.

---

## 1. The Core Problem

### 1.1 How Transformers Currently Work

A standard large language model processes every inference request the same way. The entire prompt — system instructions, conversation history, character descriptions, world state — is serialized into a flat linear sequence of tokens. Each token gets a positional encoding and the whole sequence flows through a stack of transformer blocks from input to output.

Inside each transformer block, self-attention allows every token to attend to every other token in the sequence simultaneously. This produces rich, context-saturated representations. But the fundamental constraint remains: everything that influences the model's output must occupy a position in the token sequence.

> **Key constraint:** The model's static trained parameters represent the totality of its knowledge. Every inference draws on that entire parameter space equally, weighted only by what is explicitly present in the prompt. There is no ambient context, no mood, no unconscious prior — only the token sequence.

### 1.2 The Human Cognition Contrast

Human thought does not work this way. When a person speaks, they are not attending to a word-for-word transcript of the last thirty minutes. They are influenced by a semantic cloud of unconscious context — the emotional residue of recent experiences, the ambient atmosphere of their environment, their internalized knowledge of the people they are speaking with — none of which is explicitly recalled or verbalized.

A human writer who knows their character deeply does not recite that character's backstory before writing each sentence. The knowledge is ambient. It biases word choices, sentence rhythms, and emotional register without ever becoming explicit text.

### 1.3 The Cost of the Current Approach

The standard solution is to stuff context into the prompt. Character descriptions, world state, recent events, relationship histories — all serialized into tokens and prepended to every inference call. This creates several compounding problems:

- Transformer attention complexity scales roughly quadratically with sequence length. A 4,000 token context does not cost 10x more than a 400 token context — it costs closer to 100x more in attention computation.
- The model has no way to distinguish between what is structurally important (the actual prompt) and what is reference material (the stuffed context). It attends to everything with equal positional treatment.
- The model must re-process identical context on every single inference call, with no ability to cache semantic understanding at the representation level.
- There is a hard ceiling. Context windows are finite. As story complexity grows, something always has to be dropped.

## 2. Semantic Memory Injection — The Proposed Architecture

### 2.1 Core Insight

> The attention mechanism does not require that its Key and Value matrices be derived exclusively from the current input sequence. Mathematically, Keys and Values are just matrices of vectors. They can be sourced from anywhere — including pre-computed, externally stored embeddings that were never part of the prompt.

This is not a theoretical curiosity. It is already deployed in production in a different domain: IPAdapter in diffusion-based image generation injects image embeddings into cross-attention layers as parallel Key/Value streams alongside text conditioning. The image influences the output without being a word in the prompt.

Semantic Memory Injection applies this same principle to language model self-attention layers, using the model's own embedding space to encode persistent contextual knowledge that biases generation without consuming prompt tokens.

### 2.2 Architectural Overview

The architecture separates the prompt into two distinct streams:

- **Direct Prompt Stream** — the actual input query, occupies the context window normally, flows through self-attention as usual.
- **Semantic Memory Stream** — pre-computed embeddings representing characters, world state, emotional atmosphere, relationship context. These are injected into the Key/Value matrices of selected attention layers at inference time.

The model's Query vectors — what each token is asking for — remain derived from the direct prompt. But the Key/Value pool that those queries attend to is augmented with the semantic memory stream. The model reaches for what it needs from both the prompt and the memory cloud simultaneously.

Because the semantic memory embeddings are generated by the same model's embedding space, no learned projection layer is required. The dimensional format is already compatible. The geometry is already aligned.

### 2.3 Why the Same Model's Embedding Space Matters

This is the architectural detail that makes a proof of concept achievable without model retraining. A foreign embedding — one generated by a different model or embedding system — would require a learned projection layer to translate between incompatible geometric spaces. That projection would need training.

But if the memory embeddings are generated by the same model processing the same token vocabulary, they already exist in the correct dimensional space. The model already knows how to interpret them. The injection is not introducing a foreign representation — it is reintroducing a representation the model already understands, simply through a different pathway.

### 2.4 Injection Point Options

The transformer stack offers multiple viable injection points, each with different characteristics:

#### Input Embedding Layer
Blend memory vectors directly into the input token representations before any transformer blocks run. The simplest implementation. Blunt — affects all subsequent processing uniformly. Best for coarse tonal or atmospheric influence.

#### Middle Attention Layer K/V Injection *(Recommended Starting Point)*
Intercept the Key and Value matrix computation inside selected transformer blocks — specifically the middle third of the layer stack — and augment them with memory embedding vectors. Early layers are still resolving syntax and local token relationships. Deep layers are already committed to output trajectory. The middle layers are where semantic meaning is most richly and flexibly represented. This is the most surgically precise injection point for character and world knowledge.

#### Parallel Attention Streams
Run two self-attention computations per block in parallel — one over the prompt tokens, one over the memory embeddings — then merge the resulting context vectors with a tunable blend weight before passing to the feedforward network. This is the most powerful long-term architecture and mirrors exactly how IPAdapter operates in diffusion models. Requires more scaffolding but allows the clearest separation between prompt attention and memory attention.

#### Recursive Layer Processing
Pass the output of middle layers back through earlier layers with updated memory injection before continuing the forward pass. Approximates iterative refinement — a first pass impression followed by reconsideration. Doubles compute cost for affected layers but enables a form of attention that has no equivalent in standard transformer inference.

## 3. The Semantic Memory Store

### 3.1 Memory as a Living Object

Unlike prompt-stuffed context which is regenerated on every call, semantic memory embeddings are persistent, versioned, and independently updatable. Each entity in the system — a character, a location, a relationship, an emotional atmosphere — has a corresponding embedding stored in a vector database.

This embedding is not static. It is updated as story events occur, character arcs develop, and relationships evolve. The memory is a living representation of the entity's current state in the narrative.

### 3.2 Composite Memory Injection

Multiple embeddings can be injected simultaneously with independent blend weights. For a given inference call, the memory stream might include:

- **Character embedding** — who this person fundamentally is
- **Relationship embedding** — the current dynamic between characters present
- **Scene atmosphere embedding** — the emotional and sensory register of the current moment
- **Recent events embedding** — the semantic residue of what has just occurred
- **World state embedding** — relevant facts about the broader narrative context

Each component can be weighted independently. A tense confrontation scene might heavily weight the relationship and atmosphere embeddings. A moment of quiet reflection might weight character and recent events more heavily. The orchestration layer controls these weights without modifying the model or the prompt.

### 3.3 Storage Architecture

The memory store maps naturally onto existing vector database infrastructure. Each entity record contains:

- `current_embedding` — the active semantic representation
- `embedding_history` — versioned snapshots at key narrative moments
- `last_updated` — timestamp and triggering event
- `component_type` — character, location, relationship, atmosphere, event, etc.

Retrieval at inference time is a semantic search: embed the incoming prompt, find the closest memory embeddings by cosine similarity, inject the top-N results weighted by similarity score. The orchestration layer does not need to decide which memories are relevant — the geometry of the embedding space determines relevance automatically.

## 4. Performance Implications

### 4.1 Quadratic Scaling Relief

Transformer attention scales roughly as O(n²) with sequence length. Reducing a 4,000 token prompt to 400 tokens by moving context into the memory stream does not yield a 10x speedup — it yields closer to a 100x reduction in attention computation cost for those layers.

The injected memory vectors do add some K/V computation overhead, but this overhead scales linearly with the number of injected vectors, not quadratically with sequence position. The tradeoff is strongly favorable for rich contextual knowledge.

### 4.2 KV Cache Efficiency

Shorter sequences mean reduced KV cache memory pressure. This translates directly to better GPU memory utilization, which enables larger effective batch sizes or accommodation of larger models within the same VRAM footprint. The memory constraint that currently forces a choice between context richness and model size begins to relax.

### 4.3 Pre-computation and Caching

Memory embeddings can be generated once and cached indefinitely until the underlying entity changes. A character who has not experienced any new narrative events since the last inference call costs zero additional computation — the cached embedding is simply retrieved and injected. Only updates require new embedding generation, which is itself a fast operation.

## 5. Proof of Concept — llama.cpp Implementation Path

### 5.1 Objectives

The proof of concept has a single falsifiable hypothesis: semantic memory embeddings injected into the K/V matrices of selected attention layers will measurably influence model output in a manner consistent with the semantic content of those embeddings, without that content being present anywhere in the prompt.

Success means the model's outputs drift toward the character or context represented by the injected embedding. Failure means the outputs show no systematic relationship to the injected content.

### 5.2 Implementation Steps

#### Step 1 — Identify the Attention Layer K/V Computation in llama.cpp

Locate the attention block construction in the codebase. The relevant function is likely in the neighborhood of `llm_build_kv` or the core attention tensor operations in the GGML layer. Map the data flow: where Q, K, and V tensors are computed, where attention scores are calculated, where the output is projected back.

1. Search the source for `llm_build_kv`, `ggml_mul_mat`, and attention-related function names
2. Trace the tensor shapes at each step to understand dimensionality
3. Identify which layers correspond to the middle third of the stack for the target model

#### Step 2 — Build the Memory Embedding Generator

Use the same model to generate embeddings for test entities. This can be done via llama.cpp's existing embedding mode or by extracting hidden states from a specific layer using the hidden state extraction work already completed. Generate embeddings for several test entities with clearly distinct characteristics — a stoic warrior, a whimsical trickster, a grieving scholar. Store these as flat binary float arrays.

#### Step 3 — Implement the K/V Injection Hook

Modify the attention computation to accept an optional external K/V augmentation. The injection point should concatenate or blend the memory vectors into the existing K/V tensors before the attention score computation. Start with simple concatenation — append the memory vectors to the K/V pool. The model will attend to them as additional positions.

1. Add a parameter to the attention function to accept optional external KV pairs
2. Concatenate external K vectors to the existing K tensor along the sequence dimension
3. Concatenate external V vectors to the existing V tensor along the sequence dimension
4. Ensure attention mask handling accommodates the augmented sequence length
5. Add a blend weight scalar to control injection influence

#### Step 4 — Design the Test Harness

Identify twenty to thirty questions the baseline model cannot answer from training data alone — questions about fictional entities, invented world facts, made-up character histories. Establish a clean baseline: run all questions with no injection and record outputs.

Then run two experimental conditions: injection with a semantically matching memory embedding, and injection with a semantically mismatched embedding. Three conditions total per question. Measure:

- **Factual recall** — does the model produce information consistent with the injected embedding?
- **Stylistic drift** — does the model's tone, vocabulary, and emotional register shift toward the injected entity?
- **Mismatch detection** — does injecting a mismatched embedding produce outputs inconsistent with both the prompt and the matching embedding? (This rules out random noise effects.)

#### Step 5 — Blind Evaluation

Generate twenty response pairs — one from baseline, one from injection — for neutral prompts with no objectively correct answer. Shuffle them. Present to a human rater who does not know which condition produced which response and ask them to sort the pairs into two consistent groups. If the rater can sort them reliably, the injection effect is perceptible and systematic. This is a stronger signal than factual recall alone.

### 5.3 Expected Failure Modes and Mitigations

- **No measurable effect** — blend weight too low, or injection at wrong layer depth. Sweep blend weights from 0.1 to 1.0, test injection at early, middle, and late layers independently.
- **Incoherent outputs** — blend weight too high, memory vectors overwhelming prompt attention. Reduce blend weight, consider injecting into fewer layers.
- **Effect present but weak** — model has insufficient capacity to utilize injected context without fine-tuning. This is still meaningful data. Document the observed effect size and note that fine-tuning the injection pathway would likely amplify it.
- **Dimensional mismatch errors** — embedding extraction layer does not match K/V dimensions at injection layer. Verify tensor shapes at each step before attempting injection.

## 6. Long-Term Vision and Applications

### 6.1 Decoupled Knowledge and Language

The deepest implication of this architecture is a fundamental reframing of what model scale is for. Currently, large parameter counts serve two conflated purposes: storing factual and world knowledge, and providing sophisticated language reasoning and composition capability. These do not need to be the same model.

A smaller, efficient language engine — one that deeply understands syntax, semantic flow, narrative structure, emotional register — could serve as a universal base. Domain knowledge, character knowledge, world knowledge, relationship knowledge all live externally in the vector store and are injected on demand. The model does not need to know who a character is. It needs to *feel* who a character is through the injected embedding, and then write them.

> The expertise travels with the context, not the model. The same base model, with different memory injections, becomes a fantasy novelist, a legal document drafter, a technical writer, or a therapist — not through prompt engineering, but through ambient semantic context operating at the attention level.

### 6.2 Application to Narrative AI Systems

For a system like Luna-Narrates, this architecture eliminates the preprocessing pipeline that currently makes explicit relevance decisions about what context to include in each prompt. That decision belongs to the attention mechanism, not the orchestration layer.

The simplified pipeline becomes: embed the incoming narrative prompt, semantically retrieve the closest memory embeddings from the vector store across all relevant entity types, inject them alongside the prompt into the attention layers, and let the narrator determine relevance during its own processing. The narrator is not told what to remember. It is surrounded by the semantic cloud of the story the way a human author is surrounded by their internalized knowledge of their own world.

### 6.3 Competitive Differentiation

This architecture represents a moat at the infrastructure level rather than the surface level. Surface moats — better UI, lower prices, more features — are replicated in weeks or months. An architectural moat requires a competitor to independently derive the same insight, build the same injection layer, validate the same embedding alignment approach, and rebuild their entire pipeline around it.

Additionally, operational knowledge accumulates over time in ways that cannot be copied. Every story run through a system using this architecture generates data about which injection weights, which layer depths, and which memory retrieval strategies produce the best narrative outcomes. That operational knowledge is proprietary and compounds with use.

## 7. Relationship to Prior Concepts

This architecture extends and operationalizes several concepts developed in earlier Cognitive Team architecture work:

- **Semantic Token Passthrough** — the principle of preserving meaning in compressed semantic form between processing stages rather than forcing reconstruction through natural language tokens. Semantic Memory Injection applies this principle at the model's lowest accessible level, inside the attention mechanism itself rather than at the orchestration layer.

- **Dual-Babel Storage** — the separation of compressed machine-readable semantic representations from full natural language prose. The memory embedding serves as the ultimate compressed semantic representation, operating below the token level entirely.

- **Layered Memory System** — the hierarchical organization of memory by access frequency and recency. Memory embeddings map cleanly onto this hierarchy: hot embeddings in fast cache, warm embeddings in vector database, cold snapshots in archival storage.

- **Vector Math and Associative Recall** — the use of embedding space geometry for semantic retrieval. Memory injection is the natural extension: instead of retrieving embeddings and unpacking them back into text, inject them directly into the processing that would have consumed that text.

> The Cognitive Team architecture solved the memory problem at the application layer. Semantic Memory Injection solves it at the model layer. Together they represent a complete stack — external memory management above, attention-level memory influence below — with the LLM as a pure language reasoning engine in between.

## 8. Open Questions for Exploration

1. What is the optimal layer depth for injection of different semantic content types? Character personality may inject most effectively at different depths than factual world knowledge.
2. Does blend weight need to be learned, or can it be effectively tuned through systematic experimentation without training?
3. How does the model behave when injected memory contradicts explicit prompt content? Does prompt content or memory win, and can this be controlled via blend weight?
4. Can multiple memory embeddings be combined through vector arithmetic before injection — adding character A's courage to character B's cunning to create a composite — and does the result inject coherently?
5. What is the maximum number of memory vectors that can be injected before attention dilution degrades output quality?
6. Does the approach generalize across model architectures, or are the optimal injection points architecture-specific?
